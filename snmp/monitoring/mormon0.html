It was in 1991, the Remote Network Monitoring MIB was published in RFC 1271 , known as "RMON MIB". This contains the tools needed by a network management station to configure and control a monitor , read out data and reports, and receive alarms .

There are rules to RMON in order to maintain the system in order to run. Those rules give to different operators many rights. There are systems that can give rights to one operator, or that can share them between different users (example one manage one interface and another one, the other interface of the device). It depends on choices made by vendors. Users have to select their own administrative model.
The mechanism to manage the device is the following one : the operator is identified by a variable, which is issued to implement the desired administrative model. Once control table is complete (index assigned and has valid status), the monitor starts the job required by the entry.

RFC 1271 Remote Network Monitoring Management Information Base provides many features for monitoring and a large number of general variables.

Monitor operator can configure the monitor to show data in the most significant way for the user. The operator sets the exact parameters he wants :
How long should the monitor keep the data, how long is a survey...
Two operators can ask a monitor a different survey type for the same parameter ; if the monitor can hold it, it will be done.

To make the difference between several manager's requests, there are two control table variables. One identifies who wrote the request and the other indicates the current status of the control table entry.
These variables are labelled OwnerString, it contains information about the owner of the entry :
  • the administrative name
  • IP address
  • management station name
  • operator’s name
  • location or phone number
    and EntryStatus is an integer : valid(1), createRequest(2), underCreation(3), invalid(4)). It starts with a set that create the request in the control table :
  • the request is initialised and the entry becomes valid
  • the operator create the request
  • the monitor gather information
  • the operator sets the invalid condition to end the task

    There is a large number of groups in the RMON MIB. The order hides the close relationship between groups. They are the following one :
  • statistics (1.3.6.1.2.1.16.1)
  • history (1.3.6.1.2.1.16.2)
  • host (1.3.6.1.2.1.16.4)
  • hostTopN (1.3.6.1.2.1.16.5)
  • matrix (1.3.6.1.2.1.16.6)

    The following groups are working together. They work as a whole, as a diagnostic team. They are used in case of network problems, when a particular kind of traffic needs to be watched precisely. The event group, packet group and filter group are linked to give a fine tuned of diagnostic tools.

  • filter (1.3.6.1.2.1.16.7)
  • packet capture (1.3.6.1.2.1.16.8)
  • alarms (1.3.6.1.2.1.16.3)
  • events (1.3.6.1.2.1.16.9)

    Configuration, Control, and Data Tables

    A monitor is very powerful and can do a lot, but you have to know how to ask him in order to get what you want. Thus, the configuration for this type of device is very important. This MIB contains a lot of configuration parameters.

    A control table configures the information to be monitored, for example there is the time interval for capture that must be set with the amount of data to keep and record. A separate data table holds the information that are gathered. One control entry is linked to the data that is gathered.
