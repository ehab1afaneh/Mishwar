Datatype Changes :
Basic datatypes were expanded slightly, counters and gauges will be defined with their proper implementation size (Counter32, Counter64, Gauge32) other new types were introduced : UInteger32 (unsigned Integer 32 bits long) and NsapAdress (OSI addresses).
BIT STRING datatype was added to replace the INTEGERs that express a bunch of flag settings (the BIT STRING can only be enumerated, a value is assigned to one bit and not to a bunch).

Setting Values :
Protection in updating and setting values are added. When the new value is set, it depends on the current value, the management station needs to be sure that the set operations are performed in the proper order.

Managing Rows In A Table :
SNMP v2 introduces a new textual convention that promotes MIB data integrity against simultaneous actions by multiple managers (add, modify and delete table rows). A new column with RowStatus is added to any table that can be updated. RowStatus manage the creation and deletion of rows in a table.
It is expressed by an INTEGER with 6 values (from 1 to 6) which are :
  1. active
  2. notInService (during its life, a table entry can be switched between active and notInService, for example when an update is going to be done or when a function on a device is unavailable)
  3. notReady
  4. createAndGo (the manager creates the row by sending a set message and set the value, then if the creation is correct, the value is active)
  5. CreateAndWait (when the manager does not want to use the row yet, or if some columns value are not set yet.)
  6. destroy

Module Enhancements :
A good MIB module is a module on which errors will be found and on which new features could be add.

The new MODULE-IDENTITY Macro contains information about when the module has been LAST UPDATED, the ORGANIZATION responsible for the module, the person to contact, the DESCRIPTION of the module and its revision history.
Thus a vendor can identify the release of a product and where to report bugs.
When you update a module:
  • you are allowed to add new objects (including new table columns).
  • you can mark useless old objects as deprecated, obsolete or delete them.
  • you can also revise an existing module by adding units, references, definition value or you may simply update the syntax.
